# OF-11 · Модуль «Офис-менеджер» (office)

Постпродажный документальный конвейер CMR-ERP: ведёт сделку после победы в CRM
по стадиям **отгрузка → сбор документов → оплата** в виде канбан-воронки.
В отличие от исходной заглушки, этот модуль **связан с пятью отделами** через
событийную шину, **создаёт заявку перевозчику на доставку по РБ** прямо из карточки
и реализует **полную лестницу эскалации дебиторки по ТЗ §4.2**.

- **API-префикс:** `/office`  ·  **схема БД:** `office`  ·  **версия:** `0.3.0`
- **Стек:** FastAPI · SQLAlchemy (async) · Pydantic · событийная шина `core` (outbox)
- **Превью UI:** `office-manager.html` — самодостаточный HTML-макет (design-system CMR-ERP)

## Карта связей (офис в центре контура)

**Входящие** (подписки, см. `module.register`):

| Событие | Источник | Реакция офиса |
|---|---|---|
| `sales.deal.won` | CRM/Sales | Завести документ; поставить резерв; попросить Склад собрать |
| `wms.shipment.completed` | Склад | Двигать документ в «Отгружено» |
| `logistics.delivery.delivered` | Логистика | Зафиксировать трекинг/доставку в документе |
| `finance.payment.received` | Финансы | Двигать в «Оплачено», обнулить просрочку |

**Исходящие** (эмиты, см. `events.py` / `routes.py`):

| Событие | Получатель | Когда |
|---|---|---|
| `office.doc.created` | Дашборд владельца | Документ заведён |
| `office.reservation.requested` | Склад | Сделка выиграна — держать остаток под отгрузку |
| `office.shipment.requested` | Склад | Собрать заказ к отгрузке |
| `logistics.delivery.requested` | **Логистика** | Создана заявка перевозчику (только со стадии «Готово к отгрузке») |
| `office.docs.collected` | Финансы | Стадия «Сбор документов» |
| `office.payment.awaiting` | Финансы | Стадия «Ожидают оплаты»; флаг `large_receivable` при сумме > 10 000 BYN |
| `office.payment.reminder` | Клиент (офис) | Просрочка > 5 дн |
| `office.claim.requested` | Юрист | Просрочка > 15 дн — претензия |
| `office.lawsuit.requested` | Юрист | Просрочка > 30 дн — исковое заявление |
| `office.payment.escalated` | Владелец / РОП | Просрочка > 45 дн — эскалация |

Шина — transactional outbox (`core/services/eventbus.py`): событие пишется в той же
транзакции, что и изменение БД (гарантия at-least-once), доставка подписчикам — relay.

> **Лестница эскалации** (`escalate_overdue`) в проде запускается ежедневным
> планировщиком по всем неоплаченным документам; в прототипе вычисляется при переходе
> в «Ожидают оплаты» по текущему `overdue_days`. Пороги — константы в `events.py`
> (`OVERDUE_REMINDER_DAYS` … `OVERDUE_ESCALATE_DAYS`).

> **Кредитный риск ≠ согласование платежа.** Флаг `large_receivable` — это контроль
> входящей дебиторки (РОП/Финансы). Согласование исходящих платежей > 10 000 BYN
> (матрица казначейства, ТЗ §4.2) — зона финансового модуля, не офиса.

## Заявка перевозчику на доставку по РБ

Кнопка **«Создать заявку перевозчику»** на карточке стадии «Готово к отгрузке»
открывает модалку с выбором перевозчика из справочника (`carriers.py`):

СДЭК · Европочта · Белпочта · Деловые Линии · DPD · Свой транспорт
(тип, рейтинг, срок, цена от, зона, флаг тяжёлого груза).

После выбора направления и даты забора создаётся номер заявки `ЛОГ-2026-NNNN`,
перевозчик фиксируется в документе, и в Логистику уходит событие
`logistics.delivery.requested`. Заявка возможна **только** пока документ на стадии
«Готово к отгрузке» (гард `is_ready_for_carrier`) — защита от двойной логистики.

> AI-подбор оптимального перевозчика по весу/гео/цене — **Итерация 1** (в прототипе
> отрисован как ghost-плейсхолдер «Итерация 1 · ИИ / Недоступно в прототипе»).

## API-эндпоинты

| Метод | Путь | Назначение |
|---|---|---|
| `GET` | `/office/docs` | Документы по сделкам (список) |
| `GET` | `/office/board` | Канбан-воронка по стадиям |
| `GET` | `/office/carriers` | Справочник перевозчиков по РБ |
| `POST` | `/office/docs` | Создать документ (авто-номер `ДОК-2026-NNNN`) → `office.doc.created` |
| `PATCH` | `/office/docs/{id}` | Сменить стадию → событие соседнему отделу (+ лестница эскалации на `await_pay`) |
| `POST` | `/office/docs/{id}/carrier-request` | Заявка перевозчику → `logistics.delivery.requested` (гард стадии) |

## Стадии воронки (`stages.py`, без изменений)

`ready` Готово к отгрузке · `shipped` Отгружено · `docs` Сбор документов ·
`await_pay` Ожидают оплаты · `paid` Оплачено.

## Дорожная карта улучшений (рекомендации, требуют миграции БД)

Реализуется на следующих итерациях — добавляет колонки в `OfficeDoc`:

1. **Предоплата / отсрочка** (`payment_terms`): для предоплатных клиентов оплата
   гейтит отгрузку (платёж до сборки), для отсрочки — текущий ship-then-collect.
2. **Частичная отгрузка / частичная оплата** (`shipped_qty`, `paid_amount`): B2B-опт
   часто закрывается частями; документ перестаёт быть атомарным.
3. **SLA по стадиям** (`stage_since`, `due_date`): отслеживание времени-в-стадии и
   алерты по узким местам (например, «Сбор документов» > 3 дн).
4. **Разделение документов** на отгрузочные (ТТН/CMR — до отгрузки) и закрывающие
   (ЭСЧФ ≤ 10 дн, акт сверки — на закрытии периода).
5. **Идемпотентность входящих событий** (`last_event_id`) против повторной доставки.

## Миграция БД

Колонки `region`, `weight`, `address`, `sales_ref`, `wms_ref`, `logistics_ref`,
`finance_ref`, `legal_ref`, `overdue_days` добавлены **аддитивно** с `server_default`
(безопасно для существующих строк):

```bash
alembic revision --autogenerate -m "office: связи отделов, доставка, заявка перевозчику"
alembic upgrade head
```
