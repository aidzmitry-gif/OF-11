# HANDOFF · Продолжение работы над модулем «Офис-менеджер» в Claude Code

Этот документ — точка входа для продолжения работы в VS Code / Claude Code.
Открой репозиторий в VS Code, запусти Claude Code и дай ему прочитать этот файл и
`CLAUDE.md`. Дальше — по разделу «Первый промпт для Claude Code».

---

## 1. Контекст проекта

CMR-ERP — AI-First ERP/CRM для B2B опт+производство (Минск, РБ): импорт АКБ / инверторов /
солнечных панелей / ячеек из Китая, сборка кастомных батарей, продажи по СНГ. Валюта — **BYN**.
Бэкенд: FastAPI + Python, плагины-модули в `modules/<name>/`, событийная шина (outbox),
Alembic, CI-гейт покрытия **90%**. Фронт: Next.js 15 / React 19 / TS / Tailwind.

Принципы (важно соблюдать дальше):
- **AI — всегда Итерация 1.** В прототипе ноль работающего AI; AI-элементы — ghost-плейсхолдеры
  «Итерация 1 · ИИ / Недоступно в прототипе».
- **Человек всегда на карточке** рядом с любым AI-агентом.
- **Разделение ролей.** Физические операции — склад (кладовщик/сборщик/диспетчер);
  коммерческие действия (резерв, цена) — CRM/продажи. Кнопка видна только владельцу действия.
- **Валовая прибыль важнее объёма.** KPI — про маржу, не про выручку/headcount.
- **Финансовая точность.** Масштаб BYN = реальный масштаб бизнеса. Интеграция с 1С:КА.

## 3. Что реализовано в модуле

**Связи с отделами через шину (outbox, at-least-once):**

Входящие (подписки в `module.register`):
- `sales.deal.won` → завести документ, поставить резерв, попросить Склад собрать
- `wms.shipment.completed` → стадия «Отгружено»
- `logistics.delivery.delivered` → трекинг доставки
- `finance.payment.received` → стадия «Оплачено», обнулить просрочку

Исходящие (эмиты в `events.py`/`routes.py`):
- `office.doc.created`, `office.reservation.requested` (Склад), `office.shipment.requested` (Склад)
- `logistics.delivery.requested` (Логистика, заявка перевозчику — только со стадии «Готово к отгрузке»)
- `office.docs.collected` (Финансы), `office.payment.awaiting` (Финансы, +флаг `large_receivable`)
- **Лестница просрочки дебиторки (ТЗ §4.2):** `office.payment.reminder` (>5дн) →
  `office.claim.requested` (>15дн, Юрист) → `office.lawsuit.requested` (>30дн, Юрист) →
  `office.payment.escalated` (>45дн, Владелец/РОП)

**Заявка перевозчику по РБ:** `GET /office/carriers` (СДЭК/Европочта/Белпочта/Деловые Линии/DPD/
Свой транспорт) + `POST /office/docs/{id}/carrier-request` → генерит `ЛОГ-2026-NNNN`, эмитит
`logistics.delivery.requested`. Гард: только стадия «Готово к отгрузке».

**RBAC:** права `office.doc.read|write`, `office.stage.move`, `office.carrier.request` + роль «Офис-менеджер».

## 4. Интеграция в CMR-ERP_main (первый шаг в Claude Code)

Модуль написан под живое приложение, но в `CMR-ERP_main/modules/office/` сейчас лежит **старый стаб**.
Нужно перенести файлы из OF-11 в `modules/office/` главного репозитория:

```bash
# из корня CMR-ERP_main, добавив OF-11 как remote или клонировав рядом
# скопировать: carriers.py events.py models.py schemas.py routes.py module.py
# (stages.py и __init__.py идентичны стабу — можно не трогать)
```

Затем — миграция БД (модель получила новые колонки, все аддитивные с `server_default`):

```bash
alembic revision --autogenerate -m "office: связи отделов, доставка, заявка перевозчику, эскалация"
alembic upgrade head
```

## 5. Проверка (обязательно — код не прогонялся через CI в среде сборки)

Правки аддитивные и согласованы между `events.py`/`routes.py`/`module.py`, но **тесты в этой
среде не запускались**. В Claude Code:

```bash
python -m py_compile modules/office/*.py        # синтаксис
pytest modules/office -q                          # тесты модуля
pytest --cov=modules/office --cov-report=term     # покрытие (гейт 90%)
```

Тесты, которые стоит добавить (сейчас их на новый код нет):
- `escalate_overdue`: каждая ступень 5/15/30/45 эмитит верное событие; ≤5 — ничего.
- `carrier_request`: гард 409 для не-`ready`; happy-path генерит `ЛОГ-…` и эмитит событие.
- обработчики `on_deal_won` (резерв+сборка, дедуп по `sales_ref`), `on_*` переходы стадий.
- `update_doc`: переход в `await_pay` эмитит `office.payment.awaiting` + лестницу.

## 6. Дорожная карта (приоритизировано, по разбору операционного директора)

Эти пункты **требуют новых колонок в `OfficeDoc` → миграция**, поэтому вынесены из прототипа:

1. **Предоплата / отсрочка** (`payment_terms`) — ГЛАВНОЕ. Сейчас воронка предполагает только
   «отгрузил → жди оплату». Для нового/рискового клиента работает предоплата: платёж **до**
   отгрузки. Нужен флаг, который для предоплаты гейтит отгрузку.
2. **ТТН до отгрузки** — комплаенс РБ: ТТН/CMR обязаны быть в момент отгрузки. Разделить
   отгрузочные документы (до) и закрывающие (ЭСЧФ ≤10дн, акт сверки — на закрытии).
3. **SLA по стадиям** (`stage_since`, `due_date`) — алерты узких мест (док в «Сборе документов» >3дн).
4. **Частичные отгрузки/оплаты** (`shipped_qty`, `paid_amount`) — опт часто закрывается партиями.
5. **Идемпотентность входящих событий** (`last_event_id`) против повторной доставки.
6. **Планировщик эскалации** — `escalate_overdue` должен запускаться ежедневным джобом
   (Temporal/cron) по всем неоплаченным, а не только при переходе стадии.

## 7. Спецификация UI-макета `office-manager.html` (собрать в Claude Code)

Самодостаточный HTML (CSS+JS inline, без зависимостей), под общий дизайн-система CMR-ERP
(см. макеты других модулей: CRM-LID1.1, ZAK-3 — совпасть с ними):
- **Шрифт** Golos Text. **Шелл** `grid-template-columns: 236px 1fr 320px`. Палитра — CSS-переменные.
- **Сайдбар** (модули, активен «Офис-менеджер»), **топбар** (поиск, Фильтры, кнопка «ИИ-режим»
  ghost-dashed, «+ Создать документ»).
- **Полоса связей**: CRM · сделка выиграна → ОФИС-МЕНЕДЖЕР → Склад / Логистика / Финансы / Юрист (цветные чипы).
- **KPI** (5): К отгрузке 3/5; Отгружено сегодня; Сбор документов 3·1 просрочка; Ожидают оплаты (BYN);
  Просрочка дебиторки (BYN) → Юристу.
- **Канбан 5 колонок** = stages.py. Карточка: № `ДОК-2026-NNNN`, бейдж приоритета, компания
  (формы РБ: ООО/ЧТУП/ОДО/ИП/СООО), товар (АКБ/инверторы/панели), сумма **Br**, доставка+вес,
  **ответственный человек** (обязательно), след. шаг, чипы связей (crm/wms/logi/fin/legal),
  у части — ghost-плейсхолдер AI «Итерация 1», у просроченных — чип эскалации.
- **Кнопка «Создать заявку перевозчику»** на карточках стадии «Готово к отгрузке» → модалка:
  сводка груза, выбор направления (города РБ), ghost AI-подбор, **6 перевозчиков выбираемых**
  (СДЭК/Европочта/Белпочта/Деловые Линии/DPD/Свой транспорт — тип/рейтинг/срок/цена от/зона,
  тяжёлый груз ≥300кг подсвечивает паллетных), форма (дата забора, контакт, комментарий) →
  submit генерит `ЛОГ-2026-NNNN`, метит карточку, добавляет чип Логистики, добавляет событие
  в правый рейл, тост.
- **Правый рейл «Связи и события»**: ghost ИИ-копилот (Итерация 1) + лента событий между отделами.
- **Низ**: «Итоги по конвейеру» (6 метрик, BYN).
- Стадии-переходы и события рендерятся live из JS-массива `DOCS`.

> Прежняя сборка макета (≈50 КБ, 12 карточек) валидировалась через Playwright; компактная
> версия (8 карточек) тоже рабочая. В Claude Code можно собрать заново по этой спецификации
> и прогнать `npx playwright` для проверки.
